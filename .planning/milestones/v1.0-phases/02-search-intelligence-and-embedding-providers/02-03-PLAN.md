  <action>
1. Create `DimensionValidator.cs` as an `IHostedService`:
   - Constructor DI: `QdrantClient`, `IEmbeddingService`, `IOptions<QdrantSkillsOptions>`, `ILogger<DimensionValidator>`
   - `StartAsync(CancellationToken ct)`:
     a. Check if collection exists via `client.ListCollectionsAsync(ct)` -- if collection does NOT exist, skip validation (it will be created with correct dims by CollectionInitializer)
     b. If collection exists: `var info = await client.GetCollectionInfoAsync(options.CollectionName, ct);`
        - Extract existing dimensions: `(int)info.Config.Params.Vectors.Size`
        - Compare with `embeddingService.Dimensions`
        - If mismatch: check `options.MismatchResolution`:
          - `null` (default): throw `InvalidOperationException` with message: "Collection '{collectionName}' has {existingDims}-dim vectors but provider '{providerName}' produces {providerDims}-dim. Use MismatchResolution config ('rename', 'suffix', or 'replace') to resolve."
          - `"rename"`: rename collection to `{collectionName}-old-{existingDims}` via `client.UpdateCollectionAliasesAsync` or create new + delete pattern (Qdrant doesn't support rename directly -- use the approach of creating a new collection and noting the old one is preserved). Log warning.
          - `"suffix"`: change options.CollectionName to `{collectionName}-{providerDims}` (modify the bound options). Log info.
          - `"replace"`: delete the existing collection via `client.DeleteCollectionAsync`. Log warning: "Deleted collection '{collectionName}' due to dimension mismatch (data loss)."
     c. Embedding output validation (unless `options.SkipEmbeddingOutputValidation` is true):
        - Generate test embedding: `var testVector = await embeddingService.GenerateEmbeddingAsync(options.TestEmbeddingInput, ct);`
        - Verify `testVector.Length == embeddingService.Dimensions` (sanity check the provider is consistent)
        - If collection exists: upsert a test point with payload `{ "_test": true, "name": options.TestEmbeddingKey }` and the test vector
        - If collection doesn't exist yet: skip upsert (CollectionInitializer will create it later)
   - `StopAsync`: no-op (return Task.CompletedTask)

2. Register DimensionValidator in `ServiceRegistration.cs`:
   - Add `services.AddHostedService<DimensionValidator>();` BEFORE the CollectionInitializer registration (so dimension check runs first on startup)
   - Note: IHostedService runs during host startup via `Host.RunAsync()`, which is before MCP tools become available

3. Create `DimensionValidatorTests.cs` unit tests:
   - Mock `QdrantClient` (or use a wrapper/interface if QdrantClient is not easily mockable -- if needed, create a thin `IQdrantClientWrapper` or use the approach from Phase 1 tests). Actually, QdrantClient methods are virtual/overridable -- check at implementation time. If not mockable, test the logic by extracting a testable method.
   - Test: Collection doesn't exist -> no exception, validation passes
   - Test: Collection exists, dimensions match -> no exception
   - Test: Collection exists, dimensions mismatch, no resolution -> throws InvalidOperationException with correct message
   - Test: Collection exists, dimensions mismatch, resolution="suffix" -> options.CollectionName updated
   - Test: Collection exists, dimensions mismatch, resolution="replace" -> DeleteCollectionAsync called
   - Test: SkipEmbeddingOutputValidation=true -> no test embedding generated
   - Test: Test embedding dimension mismatch (provider returns wrong-length vector) -> throws
   - If QdrantClient is not directly mockable, extract the validation logic into a testable method that takes the collection info and options as parameters. Keep DimensionValidator.StartAsync as a thin shell.
