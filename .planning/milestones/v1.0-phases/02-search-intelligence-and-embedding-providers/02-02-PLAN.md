  <action>
1. Add NuGet packages to Infrastructure.csproj:
   ```bash
   dotnet add src/QdrantSkillsMCP.Infrastructure package Microsoft.SemanticKernel.Connectors.Onnx --prerelease
   dotnet add src/QdrantSkillsMCP.Infrastructure package OllamaSharp
   dotnet add src/QdrantSkillsMCP.Infrastructure package Azure.AI.OpenAI
   ```
   Also add `<NoWarn>SKEXP0070</NoWarn>` (or the correct experimental warning code for SK ONNX connectors) to the PropertyGroup in Infrastructure.csproj. Verify the correct code at build time.

2. Create `OnnxModelResolver.cs`:
   - Static class with `static string ResolveModelPath(QdrantSkillsOptions options, ILogger logger)` and `static string ResolveVocabPath(QdrantSkillsOptions options, ILogger logger)`
   - Three-tier resolution:
     a. If `options.OnnxModelPath` is set and file exists, use it (log info: "Using custom ONNX model: {path}")
     b. Check well-known companion NuGet location: scan `AppContext.BaseDirectory` for `model.onnx` / `vocab.txt` (for future `QdrantSkillsMCP.Models.DefaultEmbedding` package)
     c. Auto-download from HuggingFace to cache dir: `Path.Combine(Environment.GetFolderPath(SpecialFolder.LocalApplicationData), "QdrantSkillsMCP", "models")`
        - Download `model.onnx` and `vocab.txt` from `https://huggingface.co/onnx-models/all-MiniLM-L6-v2-onnx/resolve/main/`
        - If `options.DisableAutoDownload` is true, throw with clear message: "ONNX model not found. Set OnnxModelPath, install QdrantSkillsMCP.Models.DefaultEmbedding, or enable auto-download."
        - Log warning when auto-downloading: "Auto-downloading ONNX model from HuggingFace..."
   - Use `HttpClient` for download (simple, no library needed)

3. Create `OnnxEmbeddingService.cs`:
   - Implements `IEmbeddingService`
   - Constructor takes `BertOnnxTextEmbeddingGenerationService` (as `ITextEmbeddingGenerationService` from SemanticKernel) and `IOptions<QdrantSkillsOptions>`
   - `Dimensions` property: return 384 (all-MiniLM-L6-v2 output dimensions). If VectorDimensions is explicitly set to non-default (not 1536 and not 384), use that.
   - `GenerateEmbeddingAsync`: use the underlying generator's `GenerateEmbeddingsAsync` method, extract the float array
   - Note: The service is created in ServiceRegistration where `BertOnnxTextEmbeddingGenerationService.Create(modelPath, vocabPath)` is called, then `.AsEmbeddingGenerator()` is registered as `IEmbeddingGenerator`. OnnxEmbeddingService wraps that generator (same pattern as OpenAiEmbeddingService).

4. Create `OllamaEmbeddingService.cs`:
   - Implements `IEmbeddingService`
   - Constructor takes `IEmbeddingGenerator<string, Embedding<float>>` and `IOptions<QdrantSkillsOptions>`
   - `Dimensions` property: use `options.VectorDimensions` (user must set this to match their Ollama model)
   - `GenerateEmbeddingAsync`: same pattern as OpenAiEmbeddingService (delegate to IEmbeddingGenerator)
   - Note: OllamaApiClient natively implements IEmbeddingGenerator, so no bridge needed

5. Create `AzureOpenAiEmbeddingService.cs`:
   - Implements `IEmbeddingService`
   - Constructor takes `IEmbeddingGenerator<string, Embedding<float>>` and `IOptions<QdrantSkillsOptions>`
   - Same implementation pattern as OpenAiEmbeddingService (they share the M.E.AI interface)
   - `Dimensions` property: use `options.VectorDimensions`

Key insight: All four services (OpenAI, ONNX, Ollama, Azure) follow the same pattern -- wrap IEmbeddingGenerator and delegate. The only differences are in how IEmbeddingGenerator is created (done in ServiceRegistration). Consider whether a single generic `EmbeddingService` class could replace all four, or keep them separate for clarity. Use your judgment -- either approach is fine, but if you go generic, keep the class name clear (e.g., `GenericEmbeddingService`).
